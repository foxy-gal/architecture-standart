| F   | **Публичный сайт**: показывать список депозитов с актуальными ставками; позволять подать заявку с **телефоном и ФИО.** | Базовый клиентский путь для новых клиентов. |

|     | **MVP-процесс**: заявку обрабатывает менеджер бэк-офиса, подтверждает условия в АБС, затем открывает депозит | Требуется стадия ручной обработки и фиксации решения в АБС. |

|     | **Управление ставками**: отказаться от XLS как “единственного источника”; обеспечить единое ведение ставок/правил, доступное бэк-офису депозитов и кредитов | Архитектурно значимо: сейчас ставки живут в XLS; нужен единый источник, права доступа, версии/даты действия. |
|     | **Трекинг статуса заявки** для клиента (создана/в обработке/ставка подтверждена/открыта/отклонена) | Снижает нагрузку на кол-центр/отделения, повышает прозрачность. |
| U   | **Единый UX**: интерфейсы сайта и ИБ соответствуют корпоративной дизайн-системе (цвета, бренд-элементы, паттерны) | Прямое требование бизнеса. |
|     | **Минимум шагов** для заявки; понятные подсказки и ошибки; объяснение “что дальше” | Важно для конверсии и аудитории старшего возраста. *(Доп. требование)* |
|     | **Доступность**: адаптивность (моб/десктоп), базовая доступность (контраст, размер шрифта, понятные тексты) | Снижает отказы и обращения в поддержку. *(Доп. требование)* |
| R   | **Доступность 24/7, 99,9%** для депозитного сервиса и онлайн-каналов | Явно задано в условии. |
|     | **Отказоустойчивость между ЦОД**: при сбое основного ЦОД сервисы ИБ/депозитов должны оставаться доступны (фейловер на резервный) | Сейчас всё в одном ЦОД — нужно закрыть разрыв с целевыми требованиями. |
|     | **Изоляция АБС**: онлайн-подача заявок не должна ставить под угрозу работоспособность АБС | Команда АБС указывает на перегрузку БД; требуется развязка/ограничение нагрузки. |
|     | **Надёжная доставка уведомлений/событий** (СМС, статусы) при частичных сбоях и повторах | Нужны идемпотентность и гарантии доставки на интеграциях. |
| P   | **“Миллисекундный” отклик UI** для просмотра/оформления (каталог, параметры, статусы) | Ожидание бизнеса; влияет на кэширование/архитектуру данных. |
|     | **Устранить текущую проблему**: справочные данные иногда грузятся > 1 секунды (как в платежах) | Требование на оптимизацию справочников/кэширование/предзагрузку. |
|     | **Горизонтальное масштабирование** веб/ИБ/новых сервисов и равномерное распределение запросов между серверами/приложениями/ЦОД | Прямое требование из условия. |
|     | **Защита от пиков**: выдерживать всплески заявок без деградации АБС (очереди/лимитирование/бекпрешер) | Вытекает из риска перегрузки АБС и требований 99,9%. *(Доп. требование)* |
| S   | **Документация**: архитектурная + интеграционная (API, события, контракты, статусы, ошибки) для дальнейшего расширения | Прямо задано: предусмотреть документацию. |
|     | **Наблюдаемость**: логи, метрики, трассировка, алёрты; мониторинг SLA/SLO (99,9%, latency) | Критично для 24/7 и быстрого поиска деградаций. *(Доп. требование)* |
|     | **Аудит и трассируемость**: кто/когда поменял ставку, кто подтвердил условия, история статусов заявки | Банковский контроль и разбор инцидентов. *(Архитектурно значимо)* |
|     | **Тестируемость и безопасные релизы**: автотесты ключевых сценариев, откат/фича-флаги, стенды | Снижает риск при развитии и возможной микросервисизации “точечно”. *(Доп. требование)* |
